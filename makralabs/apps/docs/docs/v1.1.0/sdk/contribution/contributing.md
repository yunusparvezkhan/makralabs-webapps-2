# Contributing

Makra SDK contributions are easiest to review when changes are scoped, tested, and explained with user impact in mind.

## Before you open a pull request

- Confirm the problem statement in an issue or short design note.
- Add or update tests for the behavior you changed.
- Document any new config, environment variable, or endpoint.

## Pull request checklist

1. Explain the user-facing problem.
2. Describe the implementation at a high level.
3. Call out risks, migrations, or follow-up work.
4. Include sample requests or screenshots when behavior changed.

## Review expectations

Small, focused pull requests move faster than broad refactors. If a change affects SDK contracts, include before-and-after examples to reduce reviewer guesswork.
